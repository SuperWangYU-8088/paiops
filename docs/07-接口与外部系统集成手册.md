# PaiOps 接口与外部系统集成手册

## 1. 基本约定

### 1.1 基础地址

生产入口示例：

```text
http://192.0.2.10/api
```

推荐通过 HTTPS 域名访问。

### 1.2 普通响应

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

客户端必须同时检查 HTTP 状态和 JSON `code`。`code=200` 为业务成功；401 未认证；403 无敏感管理权限；500 或其他值为业务失败。

### 1.3 用户认证

除匿名/内部专用接口外，请求头：

```http
Authorization: Bearer <accessToken>
Content-Type: application/json
```

不要把 access token 放进普通 API 查询参数。

## 2. 登录和令牌

### 2.1 登录

```http
POST /api/auth/login
Content-Type: application/json

{
  "username": "admin",
  "password": "<密码>"
}
```

成功 data 包含 accessToken、refreshToken 和 userInfo。

### 2.2 刷新

```http
POST /api/auth/refresh
Content-Type: application/json

{
  "refreshToken": "<refreshToken>"
}
```

刷新后应替换本地保存的 access 和 refresh token。

### 2.3 当前用户

```http
GET /api/auth/current
Authorization: Bearer <accessToken>
```

### 2.4 SSE 一次性票据

浏览器 `EventSource` 不能方便地设置 Authorization，因此先调用：

```http
POST /api/auth/sse-ticket
Authorization: Bearer <accessToken>
```

响应：

```json
{"code":200,"data":{"ticket":"<一次性票据>"}}
```

票据 60 秒有效且只能成功消费一次。随后连接：

```text
GET /api/executions/{executionId}/stream?ticket=<一次性票据>
```

不能缓存、复用或写入长期配置。

## 3. 告警 Webhook

### 3.1 认证

```http
X-PaiOps-Webhook-Token: <PAIOPS_ALERT_WEBHOOK_TOKEN>
```

此接口不使用用户 JWT。

### 3.2 请求

```http
POST /api/alerts/webhook
Content-Type: application/json
X-PaiOps-Webhook-Token: ...

{
  "source": "alertmanager",
  "status": "firing",
  "alerts": [
    {
      "status": "firing",
      "fingerprint": "4f78c90d",
      "labels": {
        "alertname": "HighErrorRate",
        "severity": "critical",
        "service": "order-api",
        "namespace": "production",
        "paiops_runbook_id": "12",
        "paiops_auto_execute": "true"
      },
      "annotations": {
        "summary": "order-api 5xx 比例超过 5%",
        "description": "最近 5 分钟错误率持续升高"
      },
      "startsAt": "2026-07-19T07:00:00Z",
      "endsAt": "0001-01-01T00:00:00Z"
    }
  ]
}
```

响应 data：

```json
{"processed":1}
```

`paiops_auto_execute=true` 只有在关联 Runbook 全部是 READ_ONLY 节点时生效。

### 3.3 恢复通知

同一 fingerprint 发送：

```json
{
  "source": "alertmanager",
  "status": "resolved",
  "alerts": [
    {
      "status": "resolved",
      "fingerprint": "4f78c90d",
      "labels": {"alertname":"HighErrorRate","severity":"critical"},
      "annotations": {"summary":"错误率恢复"},
      "startsAt": "2026-07-19T07:00:00Z",
      "endsAt": "2026-07-19T07:15:00Z"
    }
  ]
}
```

## 4. 告警和事件

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/alerts?status=FIRING&limit=100` | 查询告警 |
| GET | `/api/incidents?status=OPEN&limit=100` | 查询事件 |
| PATCH | `/api/incidents/{id}` | 更新状态、负责人、根因、方案、复盘 |
| POST | `/api/incidents/{id}/execute` | 从事件启动 Runbook |

更新事件：

```json
{
  "status": "CLOSED",
  "assignee": "wangyu",
  "rootCause": "连接池上限与发布后流量不匹配",
  "resolution": "回滚版本并调整连接池",
  "postmortem": "补充容量评审和发布前压测"
}
```

合法状态：OPEN、ACKNOWLEDGED、MITIGATING、VERIFYING、RESOLVED、CLOSED。

从事件启动：

```http
POST /api/incidents/31/execute
Authorization: Bearer ...
Content-Type: application/json

{
  "runbookId": 12,
  "inputData": "补充信息：刚完成 2026.07.19 版本发布",
  "idempotencyKey": "incident-31-first-mitigation"
}
```

服务端会自动注入 incidentId、title、severity、summary，不需要调用方重复拼接可信事件字段。

## 5. Runbook

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/workflows` | 创建 |
| PUT | `/api/workflows/{id}` | 更新 |
| DELETE | `/api/workflows/{id}` | 逻辑删除 |
| GET | `/api/workflows/{id}` | 详情 |
| GET | `/api/workflows` | 列表 |

请求：

```json
{
  "name": "order-api 错误率诊断",
  "description": "收集指标、日志和 K8s 状态",
  "engineType": "dag",
  "flowData": "{\"nodes\":[...],\"edges\":[...]}"
}
```

注意 `flowData` 是 JSON 字符串。服务端会递归拒绝内嵌密钥字段。凭证应使用 credentialId，模型应使用全局 configId。

## 6. 异步执行任务

### 6.1 提交

```http
POST /api/workflows/12/tasks
Authorization: Bearer ...
Content-Type: application/json

{
  "inputData": "{\"service\":\"order-api\",\"namespace\":\"production\"}",
  "idempotencyKey": "manual-order-api-20260719-001"
}
```

同一 Runbook 和幂等键重复请求返回同一 executionId。

### 6.2 查询

```text
GET /api/executions?status=RUNNING&limit=100
GET /api/executions/{executionId}
```

### 6.3 取消

```http
POST /api/executions/{executionId}/cancel
Authorization: Bearer ...
```

### 6.4 SSE

先申请 ticket，再创建 EventSource：

```javascript
const ticketResult = await fetch('/api/auth/sse-ticket', {
  method: 'POST',
  headers: { Authorization: `Bearer ${accessToken}` }
}).then(r => r.json());

const source = new EventSource(
  `/api/executions/${executionId}/stream?ticket=${encodeURIComponent(ticketResult.data.ticket)}`
);
source.onmessage = event => console.log(JSON.parse(event.data));
```

连接断开重连必须申请新 ticket。

## 7. 快照、变量与同步调试接口

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/workflows/{id}/execute` | 同步执行，适合编辑器测试，不用于正式长任务 |
| GET | `/api/workflows/{id}/executions/latest` | 最近一次执行 |
| GET | `/api/workflows/{id}/executions/{executionId}/snapshots` | 节点快照 |
| GET | `/api/workflows/{id}/executions/{executionId}/variables` | 执行变量 |
| POST | `/api/workflows/{id}/executions/{executionId}/resume` | 管理型恢复接口 |
| GET | `/api/workflows/{id}/execute/stream?ticket=...` | 编辑器 SSE 执行 |

正式运维处置优先使用异步 tasks 接口，让 Go Worker 提供锁、心跳和接管。

## 8. 审批

审批接口要求 ADMIN。

```text
GET /api/approvals?status=PENDING&limit=100
POST /api/approvals/{id}/approve
POST /api/approvals/{id}/reject
```

请求体可选：

```json
{"comment":"已核对 Dry Run、影响范围和回滚方案，同意执行"}
```

批准会在同一事务中把原任务重新入队并从审批节点继续。拒绝会把任务标为 REJECTED。

## 9. 凭证接口

要求 ADMIN。

```text
GET    /api/credentials
POST   /api/credentials
PUT    /api/credentials/{id}
DELETE /api/credentials/{id}
```

创建示例：

```json
{
  "name": "生产-Kubernetes-应用命名空间",
  "connectorType": "KUBERNETES",
  "description": "仅允许 get/patch deployments 和 scale",
  "secretData": {
    "endpoint": "https://kubernetes.default.svc",
    "bearerToken": "<token>"
  }
}
```

响应不会返回 secretData 明文，只返回元数据/秘密字段名。调用方不得依赖“读取后再原样提交”更新密钥。

## 11. Worker 内部接口

这些接口只供 Go Worker 使用，不能暴露给浏览器或第三方系统：

```text
POST /api/internal/executions/{id}/run
POST /api/internal/executions/{id}/heartbeat
GET  /api/internal/executions/stale
POST /api/internal/executions/{id}/requeue
```

头：

```http
X-PaiOps-Worker-Token: <独立内部令牌>
X-PaiOps-Worker-ID: <worker实例标识>
```

外部集成禁止使用 Worker Token。

## 12. 错误处理与重试

### 12.1 可重试

- 网络连接失败；
- 502/503/504（调用方应使用相同幂等键）；
- 后端提示暂时依赖不可用；
- SSE 断开（申请新 ticket 后重连）。

### 12.2 不应自动重试

- 400 参数错误；
- 401/403 身份或权限错误；
- 高风险审批无效；
- 出站主机不在白名单；
- Runbook 含内嵌密钥；
- 任务终态或状态冲突；
- 真实写动作返回未知结果，必须先核对外部状态。

任务创建重试必须携带稳定 idempotencyKey。不要每次重试生成新键。

## 13. 接口安全检查

- 日志中只记录 URL path，不记录 Authorization 和请求密钥；
- Webhook、Worker、用户 JWT 三种令牌不能互用；
- SSE ticket 一次性；
- 集成方不保存模型/连接器 GET 响应作为密钥备份；
- 所有调用设置连接/读取超时；
- 告警发送 resolved；
- 真实动作前核对 Dry Run 和审批；
- 通过反向代理限制内部和告警入口来源。
