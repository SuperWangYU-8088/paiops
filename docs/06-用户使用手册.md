# PaiOps 用户使用手册

## 1. 登录与界面

浏览器访问部署地址，例如：

```text
http://192.0.2.10/
```

使用应用管理员账号登录。SSH 的 `root/root` 只用于服务器维护，不是 PaiOps 应用账号，不要混用。

左侧导航主要包括：

- 运维总览；
- 告警中心；
- 事件中心；
- Runbook；
- 执行任务；
- 连接器；
- 凭证管理；
- 审批中心；
- 审计日志；
- 知识库；
- MCP 工具。

## 2. 推荐上手顺序

1. 管理员先配置连接器凭证；
2. 配置全局模型；
3. 导入 SOP 到知识库；
4. 创建纯只读诊断 Runbook；
5. 手工测试只读 Runbook；
6. 接入告警并验证事件聚合；
7. 再创建“审批 + Kubernetes 动作 + 健康验证”Runbook；
8. 先 Dry Run，确认后才真实执行。

## 3. 凭证管理

进入“凭证管理”，创建连接器凭证。

### 3.1 通用要求

- 名称要能区分环境，例如 `生产-Prometheus-只读`；
- 类型必须与节点一致；
- 描述中写清系统、环境、权限范围、负责人和到期时间；
- 不在名称或描述中写密码/Token；
- 使用最小权限账号；
- 删除前先搜索 Runbook 是否引用该 credentialId。

### 3.2 Prometheus

示例字段：

```json
{
  "endpoint": "http://prometheus:9090",
  "bearerToken": "可选"
}
```

查询账号只需读取指标，不需要管理规则或配置权限。

### 3.3 Loki

```json
{
  "endpoint": "http://loki:3100",
  "token": "可选"
}
```

### 3.4 Kubernetes

```json
{
  "endpoint": "https://kubernetes.default.svc",
  "bearerToken": "ServiceAccount Token"
}
```

建议为只读查询和写动作分别创建凭证。写动作凭证只绑定需要操作的 namespace 和 Deployment 权限，禁止使用 cluster-admin。

### 3.5 Webhook

```json
{
  "url": "https://notify.example.com/webhook",
  "authorization": "Bearer ..."
}
```

保存后页面不会重新显示明文，这是正常安全行为。

## 4. 全局模型配置

在模型配置中选择 provider，填写 API URL、模型名和 API Key。注意：

- API Key 保存后不会回填；
- 页面显示“已配置”代表服务器有密钥；
- 编辑其他字段时密钥留空会保留原值；
- 只有需要更换时才输入新密钥；
- Runbook 节点只选择全局配置，不能内嵌 apiKey；
- 不需要 AI 的 Runbook 可以完全不配置模型。

运维建议把 AI 输出视为诊断建议，关键结论必须结合指标、日志和资源查询证据。

## 5. 知识库

适合导入：

- 服务拓扑；
- 标准操作规程 SOP；
- 常见告警解释；
- 历史故障复盘；
- 变更和回滚说明；
- 联系人和责任边界（避免写敏感个人信息）。

操作：

1. 新建知识库；
2. 设置分片大小和重叠；
3. 上传文档或粘贴文本；
4. 预览分片；
5. 建立索引；
6. 用搜索功能验证关键词；
7. 在知识检索/AI 节点中选择 knowledgeBaseId。

文档更新后应重新索引。不要把密码、私钥、Token、kubeconfig 导入知识库。

## 6. Runbook 编辑器

### 6.1 基本操作

1. 进入“Runbook”；
2. 新建或打开 Runbook；
3. 从左侧节点栏拖入画布；
4. 连接节点；
5. 选中节点，在右侧填写参数；
6. 保存；
7. 使用测试执行；
8. 在“执行任务”查看实际异步任务。

运维任务必须使用 DAG 引擎。每条分支都要有明确终点；避免环路；条件字段引用应与前驱输出一致。

### 6.2 节点超时

每个节点可设置 `nodeTimeoutSeconds`。建议：

- Prometheus/Loki 查询：10–30 秒；
- HTTP 健康检查：5–20 秒；
- Kubernetes 查询/动作：10–60 秒；
- AI 诊断：60–300 秒；
- 视频等非运维长任务按实际设置，但不建议放入生产处置 Runbook。

### 6.3 只读节点

常用只读节点：

- `prometheus_query`：PromQL；
- `loki_query`：LogQL；
- `kubernetes_query`：Pod/Deployment/Event 等查询；
- `http_health_check`：HTTP 状态/内容验证；
- `host_resource_check`：主机 CPU、内存、磁盘等；
- `database_health_check`：数据库可用性/慢查询检查；
- `knowledge_retrieve`：检索 SOP；
- `llm`/`react_agent`：基于已收集证据做诊断；
- `condition`：按 severity、状态、数值阈值分支。

### 6.4 治理节点

#### 变更窗口

`change_gate` 检查当前是否允许变更，只输出 `changeGatePassed`。它不是审批，不会赋予写权限。

#### 人工审批

`manual_approval` 建议填写：

- 变更对象；
- 预期效果；
- 风险；
- Dry Run 结果；
- 回滚方式；
- 审批有效期。

任务到达后暂停，进入审批中心。审批通过后在原任务上继续。

### 6.5 Kubernetes 动作

#### 扩缩容

填写 credentialId、endpoint、namespace、Deployment、replicas。replicas 允许 0–200，生产缩到 0 属于高风险，应在审批说明中明确。

#### 滚动重启

填写 namespace 和 Deployment。动作通过修改 Pod 模板 annotation 触发滚动更新。

#### 镜像回滚

填写 namespace、Deployment、container 和目标 image。目标镜像应使用不可变 digest 或明确版本，避免 `latest`。

三个动作默认 Dry Run。流程应为：

```text
只读收集 → AI/规则诊断 → 变更窗口 → 人工审批 → K8s 动作 → 健康检查 → 输出
```

## 7. 创建第一个只读诊断 Runbook

推荐拓扑：

```text
输入
  → Prometheus 查询错误率
  → Kubernetes 查询 Deployment/Pod/Event
  → Loki 查询近期错误日志
  → 知识库检索对应 SOP
  → AI 诊断汇总
  → HTTP 健康检查
  → 输出
```

步骤：

1. 输入节点接收事件 JSON；
2. Prometheus 查询使用服务 label 参数；
3. Kubernetes 查询只读凭证；
4. Loki 时间范围控制在必要窗口；
5. 知识库 topK 建议 3–8；
6. AI 提示词要求“引用证据、区分事实和推测、不得生成执行命令”；
7. 健康检查节点输出 `healthy`；
8. 输出节点汇总查询和诊断。

保存后先手工执行，确认所有 endpoint 在白名单内。

## 8. 创建审批处置 Runbook

推荐拓扑：

```text
输入
 → 只读查询
 → 条件判断
 → 变更窗口
 → 人工审批
 → Kubernetes 重启/扩缩容/回滚
 → 等待或再次查询
 → HTTP/Kubernetes 健康验证
 → 输出
```

第一次将动作节点保持 `dryRun=true`：

1. 手工启动任务；
2. 审批时核对目标和 patch；
3. 完成后检查没有真实资源变化；
4. 再把 dryRun 改为 false；
5. 重新保存并发起新任务；
6. 审批人再次确认；
7. 执行后核对 before/after 快照；
8. 健康验证不通过时事件应回到 OPEN。

## 9. 告警中心

告警列表显示 alertName、severity、status、summary、fingerprint、关联事件和接收时间。

同一 fingerprint 的 firing 会更新现有告警并累加事件告警次数。resolved 会把告警和未关闭事件转为已解决。已经解决/关闭后再次 firing，会创建新事件，保留历史边界。

不要手工伪造 fingerprint 来合并无关服务的告警。

## 10. 事件中心

### 10.1 状态

- `OPEN`：新事件或处置/验证失败；
- `ACKNOWLEDGED`：已认领；
- `MITIGATING`：Runbook 正在处置；
- `VERIFYING`：执行成功但缺少自动健康证据；
- `RESOLVED`：告警恢复或健康验证通过；
- `CLOSED`：完成根因、处置和复盘。

### 10.2 启动 Runbook

在事件详情点击执行 Runbook：

1. 选择 Runbook；
2. 补充本次事件输入；
3. 可填写 idempotencyKey；
4. 确认启动；
5. 事件转 MITIGATING 并关联 executionId。

同一事件已有 executionId 时，再次点击会返回已有任务，避免重复处置。需要重新处置时，应明确创建新事件或经设计支持新的执行轮次，不能直接清空数据库关联。

### 10.3 关闭与复盘

关闭前填写：

- 根因：造成故障的技术/流程原因；
- 处置方案：执行了哪些步骤、结果如何；
- 复盘：发现时间、影响、时间线、有效措施、无效措施、后续行动和负责人。

只有状态转 `CLOSED` 才代表完整生命周期结束。

## 11. 执行任务中心

列表可按状态查看。点击任务打开抽屉，重点信息：

- executionId、flowId、Runbook；
- 当前状态、Worker、attempt；
- queued/start/completed/heartbeat 时间；
- 节点输入、输出、错误、耗时、重试次数；
- SSE 实时进度。

### 11.1 取消

对非终态任务点击取消：

- 队列/等待审批任务立即取消；
- 运行任务进入 CANCEL_REQUESTED，通常一秒内被引擎发现；
- 外部动作已经提交时，取消只能阻止后续步骤，不能保证撤销外部副作用；应检查 after 快照并按回滚方案处理。

### 11.2 失败处理

1. 打开节点详情定位第一个 FAILED 节点；
2. 阅读结构化错误，不只看最终摘要；
3. 检查凭证、白名单、超时和外部系统；
4. 如果已经执行写动作，先核对外部真实状态；
5. 修复配置后新建任务或按快照恢复策略处理；
6. 在事件中记录失败原因。

## 12. 审批中心

审批人必须核对：

- 事件与 executionId；
- Runbook 和审批节点名称；
- 目标环境、namespace、Deployment、容器、镜像/replicas；
- Dry Run 结果；
- 变更窗口；
- 影响范围；
- 健康验证；
- 回滚方案；
- 审批有效期。

批准后不可重复批准。拒绝应填写原因；任务会转 REJECTED。不得因请求人催促而绕过核对。

## 13. 审计日志

按操作者、动作、资源和时间查询。发生以下问题时首先查审计：

- 谁启动/取消了任务；
- 谁批准或拒绝；
- 谁修改了事件；
- 谁更改了凭证、模型或 MCP；
- 告警何时接入；
- Worker 何时开始/结束任务。

审计日志不应包含密钥；如果发现敏感值，应立即按安全事件处理并修复对应记录逻辑。

## 14. 使用禁区

- 不把 API Key、密码、Token 写进 Runbook 节点；
- 不打开自定义 MCP 命令来代替正式动作节点；
- 不为方便把 HTTP 白名单设成 `*`；
- 不给 Kubernetes 凭证 cluster-admin；
- 不把 AI 诊断当作自动恢复证据；
- 不在没有健康验证和回滚方案时批准写动作；
- 不用数据库手工把任务改 SUCCESS/审批改 APPROVED；
- 不重复点击真实动作来“试试看”。
