# PaiOps 日常运维与故障处理手册

## 1. 值班原则

1. MySQL 是最终事实，Redis 是协调层；排障时先确认数据库状态，再决定是否操作队列。
2. 不手工修改 RUNNING、WAITING_APPROVAL 或审批状态，优先使用平台 API。
3. 不在日志、工单、聊天中粘贴 `.env`、JWT、Worker Token、Webhook Token 或 Kubernetes Token。
4. 先做只读检查，再做重启；重建数据卷必须有备份和审批。
5. 真实高风险操作先 Dry Run，再审批，再执行，再健康验证。

## 2. 每日巡检

### 2.1 容器

```bash
cd /opt/paiops-src
docker compose ps
docker compose stats --no-stream
```

检查：

- 容器是否反复重启；
- mysql、redis、minio、backend 是否 healthy；
- backend/worker 内存是否持续增长；
- 磁盘是否低于 20%；
- 容器日志是否快速增长。

### 2.2 健康与入口

```bash
curl -fsS http://127.0.0.1:${PAIOPS_HTTP_PORT:-80}/api/system/health
curl -fsSI http://127.0.0.1:${PAIOPS_HTTP_PORT:-80}/
```

### 2.3 任务状态

```bash
docker compose exec -T mysql sh -c \
  'mysql -uroot -p"$MYSQL_ROOT_PASSWORD" paiagent -e \
  "SELECT status,COUNT(*) count,MIN(queued_at) oldest_queued,MAX(heartbeat_at) latest_heartbeat FROM execution_record WHERE deleted=0 GROUP BY status"'
```

关注：

- QUEUED 长时间不下降；
- RUNNING 没有新心跳；
- CANCEL_REQUESTED 超过节点超时；
- FAILED 突增；
- WAITING_APPROVAL 长期无人处理。

### 2.4 Outbox

```bash
docker compose exec -T mysql sh -c \
  'mysql -uroot -p"$MYSQL_ROOT_PASSWORD" paiagent -e \
  "SELECT status,COUNT(*) count,MIN(created_at) oldest,MAX(attempts) max_attempts FROM execution_outbox GROUP BY status"'
```

正常情况下 PENDING/PROCESSING 很少且快速归零。持续增长通常表示 Redis 连接、密码、DNS 或后端调度异常。

### 2.5 Redis 队列

```bash
docker compose exec -T redis sh -c \
  'redis-cli -a "$REDIS_PASSWORD" --no-auth-warning LLEN paiops:execution:queue'
docker compose exec -T redis sh -c \
  'redis-cli -a "$REDIS_PASSWORD" --no-auth-warning --scan --pattern "paiops:execution:processing:*"'
docker compose exec -T redis sh -c \
  'redis-cli -a "$REDIS_PASSWORD" --no-auth-warning --scan --pattern "paiops:execution:lock:*"'
```

不要直接 `DEL` processing 或 lock。恢复协程会按锁状态处理；人工删除可能造成并发执行。

### 2.6 事件与审批

```sql
SELECT severity,status,COUNT(*)
FROM ops_incident
WHERE deleted=0
GROUP BY severity,status;

SELECT status,COUNT(*),MIN(requested_at)
FROM approval_request
WHERE deleted=0
GROUP BY status;
```

## 3. 日志查看

```bash
docker compose logs --since=30m backend
docker compose logs --since=30m worker
docker compose logs --since=30m mysql
docker compose logs --since=30m redis
docker compose logs --since=30m frontend
```

实时观察：

```bash
docker compose logs -f --tail=100 backend worker
```

关键词：

- `Outbox 投递失败`：Redis 不可用或认证失败；
- `execution lock lost`：锁过期、Redis 抖动或配置错误；
- `heartbeat failed`：Worker 到 backend 网络异常；
- `abandoned execution messages requeued`：发生崩溃恢复；
- `Schema migration skipped`：旧库存在重复或迁移条件不满足；
- `凭证解密失败`：Master Key 不匹配；
- `高风险节点必须连接`：审批链缺失或 approvalId 未传递；
- `出站访问目标不在白名单`：连接器 endpoint 未授权；
- `MCP 自定义命令已禁用`：符合默认安全策略。

## 4. 常见故障

### 4.1 页面打不开

检查顺序：

```bash
docker compose ps frontend
curl -I http://127.0.0.1:${PAIOPS_HTTP_PORT:-80}/
docker compose logs --tail=100 frontend
ss -lntp | grep ':80 '
```

处理：

- 端口被占用：修改 `.env` 的 `PAIOPS_HTTP_PORT` 后重建 frontend；
- 容器未启动：先确认 backend 健康，因为 frontend 依赖 backend；
- 外部访问失败但本机正常：检查防火墙、路由、反向代理和访问控制；
- 前端 502：检查 backend 容器和 Compose 网络。

### 4.2 后端 unhealthy

```bash
docker compose logs --tail=200 backend
docker compose exec -T backend wget -q -O - http://127.0.0.1:8084/api/system/health
```

常见原因：

- MySQL 未就绪/密码错误；
- schema.sql 初次初始化失败；
- Redis 密码错误；
- JVM 内存不足；
- JWT/配置校验错误；
- Master Key 与数据库密文不匹配。

不要先删卷。先修正 `.env` 或依赖服务，再执行：

```bash
docker compose up -d backend
```

### 4.3 登录失败

检查 `APP_AUTH_DEFAULT_USERNAME/PASSWORD` 是否在首次启动时正确配置。默认账户的使用方式以当前实现为准，不要把 SSH 的 root 密码当成应用管理员密码。

```bash
docker compose logs --tail=100 backend | grep -E '默认管理员|JWT_SECRET|弱密码'
```

如果已创建数据库用户，修改默认环境变量不一定等同于重置已有账户；应通过受控的账户管理/数据库恢复流程处理，禁止在工单中传明文密码。

### 4.4 任务一直 QUEUED

```bash
docker compose ps worker redis backend
docker compose logs --tail=200 worker
docker compose logs --tail=200 backend | grep -i outbox
```

分别判断：

- Outbox PENDING：后端到 Redis 故障；
- Redis 主队列有消息，Worker 无日志：Worker 配置/Token/Redis 故障；
- processing 有消息且存在 lock：任务正在执行，查看任务心跳；
- processing 有消息但无 lock：恢复协程应在 30 秒左右重排；
- 主队列为空、Outbox SENT、任务仍 QUEUED：检查 processing 和毒消息日志。

修复依赖后优先等待自动恢复。不要手工重复提交同一高风险任务；如必须提交，使用相同 idempotencyKey。

### 4.5 任务一直 RUNNING

```sql
SELECT id,flow_id,status,worker_id,heartbeat_at,started_at,attempt
FROM execution_record
WHERE status IN ('RUNNING','CANCEL_REQUESTED')
ORDER BY id;
```

- heartbeat 正常：节点仍在执行，查看任务详情和节点快照；
- heartbeat 已过期：恢复协程会重排；
- lock 正常但 heartbeat 失败：Worker 到 backend 通路异常；
- 高风险动作外部系统超时：等待节点 timeout，必要时请求取消；
- 超过 timeout 仍不结束：检查新节点是否吞掉线程中断，属于代码缺陷。

### 4.6 Outbox 堆积

```sql
SELECT id,execution_id,status,attempts,next_attempt_at,last_error,created_at
FROM execution_outbox
WHERE status <> 'SENT'
ORDER BY id
LIMIT 100;
```

处理步骤：

1. 检查 Redis healthy、密码和 DNS；
2. 检查 backend 是否多实例时间不一致；
3. 恢复 Redis 后等待指数退避；
4. 确认 PENDING 下降；
5. 不要直接把记录改 SENT，除非已经核实 Redis 存在对应消息且有变更审批。

### 4.7 Worker 反复重启

```bash
docker inspect $(docker compose ps -q worker) --format '{{.State.ExitCode}} {{.State.Error}}'
docker compose logs --tail=200 worker
```

重点检查：

- `PAIOPS_WORKER_TOKEN` 是否为空；
- Redis 地址/密码；
- backend 健康；
- 容器是否 OOMKilled；
- 并发是否超过服务器能力。

降低并发：

```dotenv
PAIOPS_WORKER_CONCURRENCY=1
```

然后：

```bash
docker compose up -d --force-recreate worker
```

### 4.8 Redis 故障

Redis 故障期间：

- 新任务仍能写 MySQL 和 Outbox，但不能投递；
- 已在执行的任务锁续租失败会主动取消控制面调用；
- SSE ticket 无法签发/消费；
- Redis 恢复后 Outbox 和 processing 恢复机制继续工作。

```bash
docker compose logs --tail=200 redis
docker compose restart redis
docker compose ps redis
```

不要随意删除 AOF 或 redis_data 卷。

### 4.9 MySQL 故障

```bash
docker compose logs --tail=200 mysql
docker compose exec -T mysql sh -c 'mysqladmin ping -uroot -p"$MYSQL_ROOT_PASSWORD"'
df -h
```

数据库不可用时停止新的处置动作。恢复后核对 RUNNING、Outbox、审批和事件状态。磁盘满时先扩容/清理已确认的旧镜像和日志，不能删除 MySQL 数据目录。

### 4.10 凭证解密失败

症状：连接器/模型执行报 `凭证解密失败，请检查 PAIOPS_MASTER_KEY`。

处理：

1. 立即停止保存新凭证；
2. 对比当前 `.env` 与备份的 key 版本，不输出值；
3. 恢复与数据库匹配的 `PAIOPS_MASTER_KEY`；
4. 重启 backend；
5. 测试一个只读连接器；
6. 如果 key 已丢失，只能重新录入相关凭证。

### 4.11 告警收不到

```bash
docker compose logs --tail=200 backend | grep -E 'ALERT_INGEST|Webhook Token'
```

用测试请求验证，令牌从安全环境变量读取，不直接写命令历史：

```bash
read -r -s -p 'Webhook Token: ' HOOK_TOKEN; echo
curl -i -H "X-PaiOps-Webhook-Token: ${HOOK_TOKEN}" \
  -H 'Content-Type: application/json' \
  -d '{"status":"firing","source":"manual-test","alerts":[{"status":"firing","fingerprint":"manual-test-001","labels":{"alertname":"PaiOpsManualTest","severity":"warning"},"annotations":{"summary":"手工接入测试"}}]}' \
  http://127.0.0.1:${PAIOPS_HTTP_PORT:-80}/api/alerts/webhook
unset HOOK_TOKEN
```

验证后发送相同 fingerprint 的 resolved 请求，避免留下无效事件。

### 4.12 高风险节点被拒绝

检查：

- 前面是否有人工审批节点；
- 任务是否在原 executionId 上续跑；
- 审批是否属于同一 executionId；
- 审批是否 APPROVED、reviewedAt 非空且未过期；
- approvalId 是否从审批节点输出传到动作节点；
- Runbook 是否在审批后被编辑，导致节点连接变化。

禁止通过手工写 `approved=true` 或数据库直接改状态绕过。

### 4.13 外部地址被白名单拒绝

先核实 URL 的主机名和业务必要性，再修改 `.env`：

```dotenv
PAIOPS_HTTP_ALLOWED_HOSTS=prometheus,loki,kubernetes.default.svc,new-host.example.com
```

```bash
docker compose up -d --force-recreate backend
```

不要为图方便设为 `*`，不要加入 `169.254.169.254` 等云元数据地址。

## 5. 备份与保留

建议：

- MySQL 每日全量、关键变更前即时备份，至少保留 7–30 天；
- `.env` 和 master key 加密离线备份；
- MinIO 根据数据价值做镜像/快照；
- 审计日志按合规要求保留；
- SENT Outbox 可在确认保留期后清理，PENDING/PROCESSING 不得清理；
- Docker 构建缓存和旧镜像按磁盘阈值清理，先检查引用。

清理 SENT Outbox 示例（先统计再执行）：

```sql
SELECT COUNT(*) FROM execution_outbox
WHERE status='SENT' AND sent_at < NOW() - INTERVAL 30 DAY;

DELETE FROM execution_outbox
WHERE status='SENT' AND sent_at < NOW() - INTERVAL 30 DAY
LIMIT 1000;
```

分批执行，观察数据库负载。

## 6. 应急停机

只停止执行面，保留查询入口：

```bash
docker compose stop worker
```

这会阻止新任务被领取，但正在 Java 内执行的 HTTP 请求可能需等待中断/超时。需要完全停止写操作时：

```bash
docker compose stop frontend worker backend
```

MySQL、Redis、MinIO 保持运行以便取证和备份。

## 7. 恢复后检查

- 容器健康；
- 数据库可查；
- Outbox 下降；
- processing 无长期无锁消息；
- RUNNING 心跳恢复；
- 没有两个 Worker 同时记录同一 executionId；
- 失败事件回到 OPEN，不被错误关闭；
- 审计记录包含恢复操作；
- 对外系统没有重复高风险副作用。
