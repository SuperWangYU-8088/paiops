# PaiOps 部署、升级与回滚手册

## 1. 适用范围

本文适用于使用 Docker Compose 部署 PaiOps。当前交付服务器为 `192.0.2.10`，代码目录为 `/opt/paiops-src`。所有命令默认以具有 Docker 权限的管理员执行。

第一次接触项目、需要从空白 Ubuntu 安装 Docker、上传交付目录并生成全部配置时，请先按 `10-从零到一部署实战.md` 操作；本文重点补充生产升级、备份和回滚。

## 2. 部署拓扑与端口

| 服务 | 容器端口 | 宿主机暴露 | 说明 |
|---|---:|---:|---|
| frontend | 80 | `${PAIOPS_HTTP_PORT}`，默认 80 | 唯一需要对用户开放的入口 |
| backend | 8084 | 不暴露 | 仅 Compose 网络访问 |
| mysql | 3306 | 不暴露 | 最终事实库 |
| redis | 6379 | 不暴露 | 队列、锁、SSE 票据 |
| minio | 9000/9001 | 不暴露 | 对象存储和控制台 |
| worker | 无监听端口 | 不暴露 | 消费任务 |

前端 Nginx 把 `/api/` 代理到 backend，把 `/storage/` 代理到 MinIO。

## 3. 前置条件

### 3.1 推荐资源

- Linux x86_64；
- 4 核 CPU，最低 2 核；
- 8 GiB 内存，最低 4 GiB；
- 30 GiB 可用磁盘；
- Docker Engine 24+；
- Docker Compose v2；
- 可访问所需镜像仓库；
- 时区 Asia/Shanghai 或明确的 NTP 同步。

### 3.2 检查

```bash
docker version
docker compose version
df -h /opt /var/lib/docker
free -h
timedatectl status
```

如果服务器不能访问公网，应预先在联网环境构建并导出六个镜像，或配置企业镜像代理。

## 4. 首次部署

### 4.0 国内镜像与依赖加速

项目已在 Compose/Dockerfile 中提供可覆盖的构建参数：

```dotenv
DOCKER_HUB_MIRROR=m.daocloud.io/docker.io
MAVEN_MIRROR_URL=https://maven.aliyun.com/repository/public
NPM_REGISTRY=https://registry.npmmirror.com
GOPROXY=https://goproxy.cn,direct
ALPINE_MIRROR=https://mirrors.aliyun.com/alpine
```

- Docker Hub 基础镜像通过 DaoCloud 前缀拉取；
- Maven 只把 Central 替换为阿里云 public，Spring Milestone 仍访问项目声明地址；
- npm 使用 npmmirror；
- Go 使用 goproxy.cn，未命中时 direct。

参考：

- [DaoCloud 镜像加速站](https://docs.daocloud.io/community/mirror)
- [阿里云 Maven 镜像配置](https://developer.aliyun.com/article/1593152)
- [npmmirror 使用说明](https://npmmirror.com/)
- [Goproxy.cn 使用说明](https://goproxy.cn/)

企业生产环境优先替换为内部 Harbor/Nexus/Artifactory/npm/Go Proxy，并实施镜像签名、漏洞扫描和准入策略。公共镜像不可用时，可临时把 `DOCKER_HUB_MIRROR` 改回 `docker.io`；不要同时修改镜像 tag，避免把网络切换和版本升级混为一次变更。

### 4.1 准备代码

```bash
cd /opt/paiops-src
test -f compose.yaml
test -f backend/Dockerfile
test -f frontend/Dockerfile
test -f worker/Dockerfile
```

不要从不可信来源覆盖此目录。部署前查看版本和变更：

```bash
git status --short
git log -1 --oneline
```

### 4.2 创建生产环境文件

```bash
cd /opt/paiops-src
cp .env.deploy.example .env
chmod 600 .env
```

编辑 `.env`，替换所有 `CHANGE_ME`。推荐分别生成，不要复用：

```bash
openssl rand -base64 36
```

必填项：

```dotenv
MYSQL_ROOT_PASSWORD=<随机数据库密码>
REDIS_PASSWORD=<随机 Redis 密码>
MINIO_ACCESS_KEY=paiopsadmin
MINIO_SECRET_KEY=<随机 MinIO 密码>
JWT_SECRET=<至少 32 字符随机值>
PAIOPS_MASTER_KEY=<至少 32 字符随机值，必须长期备份>
PAIOPS_WORKER_TOKEN=<独立随机内部令牌>
PAIOPS_ALERT_WEBHOOK_TOKEN=<独立随机告警令牌>
APP_AUTH_DEFAULT_USERNAME=admin
APP_AUTH_DEFAULT_PASSWORD=<随机管理员密码>
PAIOPS_HTTP_ALLOWED_HOSTS=prometheus,loki,kubernetes.default.svc,实际需要的域名
PAIOPS_WORKER_CONCURRENCY=2
PAIOPS_HTTP_PORT=80
```

检查是否还有占位符，只输出行号而不显示密钥值：

```bash
if grep -q 'CHANGE_ME' .env; then echo '仍有 CHANGE_ME，禁止部署'; exit 1; fi
test "$(stat -c '%a' .env)" = 600
```

### 4.3 校验 Compose

`docker compose config` 会展开密钥，不要把完整输出复制到工单或聊天。只检查返回码：

```bash
docker compose config >/dev/null
echo $?
```

### 4.4 构建

```bash
docker compose build --pull
```

构建阶段会执行：

- backend：Maven 3.9.9 + Java 21，生成 JAR；
- worker：Go 1.24，生成静态二进制；
- frontend：Node 22，执行 TypeScript 检查和 Vite 生产构建。

构建前的独立测试命令：

```bash
cd /opt/paiops-src/backend
export JAVA_HOME=/usr/lib/jvm/java-21-openjdk-amd64
mvn test
```

如果宿主机没有 Go/Node，可由 CI 或开发机执行 `go test ./...`、`npm run build`，生产镜像仍由 Docker 多阶段构建完成。

### 4.5 启动

```bash
cd /opt/paiops-src
docker compose up -d
docker compose ps
```

启动依赖顺序：MySQL/Redis/MinIO 健康 → backend 健康 → worker/frontend 启动。

### 4.6 首次健康检查

```bash
curl -fsS http://127.0.0.1:${PAIOPS_HTTP_PORT:-80}/api/system/health
docker compose ps
docker compose logs --tail=100 backend
docker compose logs --tail=100 worker
```

预期健康响应中包含 `"UP"`。所有容器应为 `running`，有健康检查的容器应为 `healthy`。

### 4.7 登录检查

不要在共享 shell 历史中直接写密码。交互读入：

```bash
read -r -p '管理员账号: ' PAI_USER
read -r -s -p '管理员密码: ' PAI_PASS; echo
curl -fsS -H 'Content-Type: application/json' \
  -d "{\"username\":\"${PAI_USER}\",\"password\":\"${PAI_PASS}\"}" \
  http://127.0.0.1:${PAIOPS_HTTP_PORT:-80}/api/auth/login
unset PAI_USER PAI_PASS
```

验证后关闭终端滚屏共享，避免令牌被旁观者看到。

## 5. 反向代理与 HTTPS

生产环境建议在 PaiOps 前增加已有 Nginx/Ingress，TLS 终止后代理到宿主机端口。至少设置：

```nginx
location / {
    proxy_pass http://192.0.2.10:80;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}

location /api/ {
    proxy_pass http://192.0.2.10:80;
    proxy_http_version 1.1;
    proxy_buffering off;
    proxy_read_timeout 1800s;
}
```

公网部署必须启用 HTTPS、限制源 IP、接入 WAF/访问日志和统一身份认证。当前默认管理员模型适用于个人或受控内网，不等同于完整企业 IAM。

## 6. Alertmanager 接入

Alertmanager receiver 示例：

```yaml
receivers:
  - name: paiops
    webhook_configs:
      - url: http://192.0.2.10/api/alerts/webhook
        send_resolved: true
        http_config:
          authorization:
            type: Bearer
            credentials: unused
```

PaiOps 要求自定义头 `X-PaiOps-Webhook-Token`。如果 Alertmanager 版本/配置不能设置自定义头，建议通过受控 Nginx location 注入该头，并仅允许 Alertmanager 源 IP访问：

```nginx
location = /paiops-alert-webhook {
    allow 192.0.2.20;
    deny all;
    proxy_set_header X-PaiOps-Webhook-Token <告警令牌>;
    proxy_pass http://192.0.2.10/api/alerts/webhook;
}
```

不要把令牌提交到公共仓库。路由规则在告警 label/annotation 中增加 `paiops_runbook_id`，只读自动执行还需 `paiops_auto_execute=true`。

## 7. 升级前备份

### 7.1 维护窗口

升级前停止新增高风险任务，等待 RUNNING/MITIGATING 任务完成。检查：

```bash
docker compose exec -T mysql sh -c \
  'mysql -uroot -p"$MYSQL_ROOT_PASSWORD" -N -e "SELECT status,COUNT(*) FROM paiagent.execution_record GROUP BY status"'
```

### 7.2 MySQL 备份

```bash
install -d -m 700 /opt/paiops-backup
BACKUP_TS=$(date +%Y%m%d-%H%M%S)
docker compose exec -T mysql sh -c \
  'mysqldump -uroot -p"$MYSQL_ROOT_PASSWORD" --single-transaction --routines --triggers paiagent' \
  | gzip > "/opt/paiops-backup/paiagent-${BACKUP_TS}.sql.gz"
test -s "/opt/paiops-backup/paiagent-${BACKUP_TS}.sql.gz"
```

### 7.3 配置与版本备份

```bash
cp -p .env "/opt/paiops-backup/env-${BACKUP_TS}"
chmod 600 "/opt/paiops-backup/env-${BACKUP_TS}"
git rev-parse HEAD > "/opt/paiops-backup/git-${BACKUP_TS}.txt"
docker compose images > "/opt/paiops-backup/images-${BACKUP_TS}.txt"
```

MinIO 对象如包含重要文件，应使用 `mc mirror` 或存储层快照单独备份。Redis 可从队列恢复协调状态，但升级前仍建议保留 appendonly 数据卷快照。

## 8. 常规升级

```bash
cd /opt/paiops-src
git status --short
# 放入经过评审的新代码后：
docker compose build
docker compose up -d
docker compose ps
curl -fsS http://127.0.0.1:${PAIOPS_HTTP_PORT:-80}/api/system/health
```

应用启动时 `SchemaMigrationRunner` 会幂等补充旧库缺失的表和列。生产 `schema.sql` 只用于首次创建卷，不会在已有 MySQL 数据卷上自动重放。

### 8.1 幂等唯一索引升级

如果旧库存在重复 `idempotency_key`，自动迁移会跳过唯一索引并记录 ERROR，避免擅自改历史数据。检查：

```sql
SELECT flow_id, idempotency_key, deleted, COUNT(*) AS duplicate_count
FROM execution_record
WHERE idempotency_key IS NOT NULL
GROUP BY flow_id, idempotency_key, deleted
HAVING COUNT(*) > 1;
```

由负责人确认保留哪条历史记录后，可以把其余重复记录的 idempotency_key 改为唯一的历史值，例如追加 `-legacy-<id>`，再执行：

```sql
ALTER TABLE execution_record
ADD UNIQUE KEY uk_execution_idempotency (flow_id, idempotency_key, deleted);
```

执行前必须备份数据库。

## 9. 回滚

### 9.1 仅应用回滚

适用于数据库结构向后兼容且没有不可逆数据写入：

```bash
cd /opt/paiops-src
docker compose down
# 恢复上一版本代码或镜像标签
docker compose up -d
```

不要使用 `docker compose down -v`，它会删除 MySQL、Redis、MinIO 数据卷。

### 9.2 数据库回滚

只有新版本已经写入旧版本不能识别的数据，或迁移失败且无法修复时才恢复数据库。步骤：

1. 停止 frontend、worker、backend，防止继续写入；
2. 保留当前故障库备份，便于取证；
3. 重新创建目标数据库或在隔离实例恢复；
4. 导入升级前 SQL 备份；
5. 恢复匹配版本代码和 `.env`；
6. 启动并做健康、登录、任务只读验证。

示例：

```bash
docker compose stop frontend worker backend
gunzip -c /opt/paiops-backup/paiagent-<时间>.sql.gz | \
  docker compose exec -T mysql sh -c \
  'mysql -uroot -p"$MYSQL_ROOT_PASSWORD" paiagent'
docker compose up -d backend worker frontend
```

### 9.3 Master Key 回滚

如果误改 `PAIOPS_MASTER_KEY`，不要重新保存任何凭证。立即停止 backend，恢复与数据库密文对应的旧 `.env`，再启动并验证凭证。新旧 key 混用期间保存的数据需要专项迁移，不能靠多次重启解决。

## 10. 升级后验收

- [ ] `docker compose ps` 全部正常；
- [ ] 健康接口 UP；
- [ ] 管理员登录成功；
- [ ] 告警列表、事件列表、Runbook、任务、审批、审计页面可打开；
- [ ] 模型配置查询不回显 API Key；
- [ ] 创建只读任务后 Go Worker 能领取并完成；
- [ ] SSE 能实时显示节点进度；
- [ ] Dry Run Kubernetes 动作输出 before/patch，不做真实变更；
- [ ] 无审批的真实高风险动作被拒绝；
- [ ] Outbox 没有持续堆积；
- [ ] 日志无 Schema migration error、lock lost 风暴或持续 heartbeat failed。

## 11. 卸载

仅停止并保留数据：

```bash
docker compose down
```

删除数据卷属于不可恢复操作，必须经过数据所有者书面确认并先完成备份。不要在日常维护中执行 `docker compose down -v`。
