# 详细设计说明书：安全治理与事件闭环

## 1. 安全目标

PaiOps 会访问监控系统、Kubernetes、数据库和模型服务，其中部分操作具有生产写权限。安全设计遵循：默认拒绝、最小权限、凭证不出库、AI 与写权限隔离、所有写动作可追溯、执行成功不等于业务恢复。

## 2. 威胁模型

| 威胁 | 典型方式 | 主要控制 |
|---|---|---|
| 密钥泄漏 | GET 配置回显、Runbook JSON 内嵌、日志打印 | AES-GCM、接口脱敏、保存前递归拦截、审计脱敏 |
| 审批绕过 | 输入 `approved=true`、伪造变更窗口 token | 高风险节点回查数据库审批事实 |
| 任意命令执行 | 自定义 MCP command/args、AI 自主工具调用 | 自定义命令默认关闭、预设完整匹配、无通用 Shell 节点 |
| SSRF | 节点访问 127.0.0.1、云元数据、恶意重定向 | 协议和主机白名单、禁用自动重定向 |
| 重复动作 | 队列重复、响应丢失、Worker 接管 | 幂等键、执行锁、状态校验、快照 |
| 权限过大 | 共用管理员凭证、Worker 持数据库密码 | ADMIN 边界、独立 Worker Token、连接器最小权限 |
| JWT 泄漏 | SSE 查询参数进入代理日志 | 一次性 60 秒 SSE ticket |
| 告警伪造 | 任意人调用 Webhook | 独立告警 Token、常量时间比较 |
| 数据库密文不可恢复 | master key 丢失/误换 | 部署后离线备份 `PAIOPS_MASTER_KEY` |

## 3. 身份与权限

### 3.1 用户 JWT

登录成功签发 access token 和 refresh token。access token 默认 120 分钟，refresh token 默认 168 小时。令牌包含 username、type 和 role。当前单机部署默认账户签发 `ADMIN`，为后续企业身份源保留角色扩展点。

受保护路径为 `/api/**`，以下路径使用专门边界或允许匿名：

- `/api/auth/login`、`/api/auth/refresh`；
- `/api/system/health`；
- `/api/alerts/webhook`，由 Webhook Token 保护；
- `/api/internal/**`，由 Worker Token 保护；
- 节点类型、OpenAPI 文档等公开元数据。

敏感管理路径要求 ADMIN：

- `/api/credentials/**`；
- `/api/mcp-tools/**`；
- `/api/llm-config/**`；
- `/api/approvals/**`。

### 3.2 Worker 身份

Worker 使用 `X-PaiOps-Worker-Token` 和 `X-PaiOps-Worker-ID`。Token 与 JWT、告警 Token 分离。Java 使用常量时间比较；未配置或不匹配时返回 401。

Worker 容器不注入 MySQL、MinIO、模型或 Kubernetes 凭证，只注入 Redis 密码和控制面 Token。

### 3.3 告警身份

告警请求头：

```http
X-PaiOps-Webhook-Token: <PAIOPS_ALERT_WEBHOOK_TOKEN>
```

Webhook Token 只能提交告警，不能直接调用用户 API。自动执行进一步受只读 Runbook 静态风险检查约束。

## 4. 凭证与密钥

### 4.1 加密格式

应用使用 `PAIOPS_MASTER_KEY` 派生 AES 密钥，采用 AES-GCM 加密，密文带版本前缀：

```text
enc:v1:<base64 nonce+ciphertext+tag>
```

GCM 同时提供机密性和完整性。随机 nonce 保证相同明文产生不同密文。读取旧版本明文时保留兼容，但下一次保存会转为新格式。

### 4.2 模型配置

- 保存 API Key 时加密；
- GET 列表和详情把 `apiKey` 设为 null；
- 返回 `apiKeyConfigured=true/false` 表示是否已有密钥；
- 编辑时密钥框永不回填；留空表示保留原密钥；
- 节点执行只能引用全局模型配置，不再接受节点内联 apiKey。

### 4.3 连接器凭证

`connector_credential.encrypted_payload` 保存整体 JSON 密文。业务接口返回元数据，不返回解密 payload。只有执行节点在服务端按 ID 获取运行时凭证。

建议的连接器类型：

- `PROMETHEUS`：endpoint、可选 bearerToken；
- `LOKI`：endpoint、可选 token/basic；
- `KUBERNETES`：endpoint、bearerToken、可选 caCert/insecureSkipTlsVerify；
- `DATABASE`：只读 JDBC/HTTP 检查凭证；
- `WEBHOOK`：url、authorization。

### 4.4 主密钥生命周期

1. 首次部署生成至少 32 字节随机值；
2. 保存于服务器 `.env`，权限 600；
3. 另做离线安全备份；
4. 不写 Git、不写文档、不复制到普通聊天；
5. 更换前必须实现批量解密重加密流程，不能直接改环境变量；
6. 丢失后已有凭证无法恢复，只能重新录入。

## 5. Runbook 密钥策略

保存和更新 Runbook 时递归检查所有节点数据。字段名忽略大小写，拒绝：

- `apiKey`、`api_key`；
- `password`；
- `secret`；
- `token`、`accessToken`、`bearerToken`；
- header JSON 中的 `Authorization`。

设计目的不是猜测所有密钥，而是建立“Runbook 只能引用 credentialId/configId，不能携带秘密”的硬约束。新增节点如需凭证，必须接入凭证服务，不能扩大例外列表。

## 6. 出站网络安全

### 6.1 允许协议

默认只允许 HTTP/HTTPS。节点不能通过 `file:`、`ftp:`、`jar:` 等协议读取本地或内部资源。

### 6.2 主机白名单

`PAIOPS_HTTP_ALLOWED_HOSTS` 是逗号分隔的允许列表，可写精确主机或受控通配符，例如：

```text
prometheus,loki,kubernetes.default.svc,monitor.example.com,*.ops.example.com
```

生产环境不建议配置 `*`。添加地址前应确认：

- 目标是业务必要系统；
- DNS 由可信网络控制；
- 不指向云元数据、Docker socket 或管理平面；
- 凭证权限为只读或动作所需最小集合。

### 6.3 重定向

JDK HttpClient 设置 `Redirect.NEVER`。原因是初始 URL 可能在白名单，但 30x 目标可能是内网或云元数据。节点如确需重定向，应显式把最终主机加入白名单并直接使用最终 URL。

## 7. MCP 与命令边界

stdio MCP 本质上会启动本地进程，是远程命令执行边界。默认：

```text
PAIOPS_MCP_ALLOW_CUSTOM_COMMAND=false
```

安全策略：

- 预设记录不可被修改为自定义命令；
- 执行前重新校验 command、args、transport 和 env 与代码内受信任预设完全一致；
- env 中的密钥加密存储；
- 不提供通用 Shell 工具给 ReAct；
- 生产开启自定义命令前必须进行镜像固定、二进制校验、参数白名单、容器只读文件系统、网络隔离和审计评审。

## 8. 审批模型

### 8.1 审批事实

审批单关键字段：executionId、flowId、nodeId、riskLevel、status、requestedBy、reviewedBy、reason、comment、requestedAt、reviewedAt、expiresAt。

合法审批必须同时满足：

- 记录存在；
- executionId 与当前执行完全相同；
- status 为 `APPROVED`；
- reviewedAt 非空；
- expiresAt 为空或晚于当前时间。

### 8.2 防伪造

工作流输入中的 `approved=true`、`changeGatePassed=true` 都不是权限事实。高风险节点执行前调用 `ApprovalService.assertValidApproval` 回查数据库。

`change_gate` 只表示当前在允许变更窗口内，输出 `changeGatePassed`，不输出 approved，也不能替代人工审批。

### 8.3 并发保护

批准/拒绝使用条件更新：

```sql
UPDATE approval_request
SET status = ...
WHERE id = ? AND status = 'PENDING';
```

受影响行数不是 1 就说明已被他人处理。审批与任务续跑在同一事务内，续跑消息提交后再通过 Outbox 投递。

## 9. Kubernetes 高风险动作

### 9.1 支持动作

| 节点类型 | 动作 | API 方式 |
|---|---|---|
| `kubernetes_scale` | Deployment 扩缩容 | PATCH scale 子资源 |
| `kubernetes_restart` | 滚动重启 | PATCH pod template annotation |
| `kubernetes_rollback` | 回滚/设置容器镜像 | PATCH deployment containers |

### 9.2 明确不支持

- 任意 Kubernetes JSON Patch；
- 删除 namespace/PVC/Secret；
- exec 进入 Pod；
- 通过节点输入传 kubeconfig；
- 拼接 shell 调用 kubectl。

### 9.3 校验

- namespace、Deployment、container 必须符合 Kubernetes DNS/名称格式；
- replicas 范围 0–200；
- timeout 范围 1–60 秒；
- endpoint 必须通过出站白名单；
- credentialId 必须存在；
- 默认 `dryRun=true`；
- 非 Dry Run 必须有真实审批；
- 执行前 GET 保存 beforeSnapshot；执行后再次 GET 保存 afterSnapshot；
- 输出 changeId 关联审计。

### 9.4 最小 RBAC 示例

建议为 PaiOps 单独创建 ServiceAccount，并按目标 namespace 绑定，不授予 cluster-admin：

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: paiops-deployment-operator
  namespace: application
rules:
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "patch"]
  - apiGroups: ["apps"]
    resources: ["deployments/scale"]
    verbs: ["get", "patch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: paiops-deployment-operator
  namespace: application
subjects:
  - kind: ServiceAccount
    name: paiops
    namespace: application
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: Role
  name: paiops-deployment-operator
```

## 10. 告警与事件聚合

### 10.1 指纹

优先使用 Alertmanager 提供的 fingerprint；缺失时对 alertName + labels JSON 做 SHA-256。`ops_alert.fingerprint` 唯一。

### 10.2 重复与重开

- 同一 firing 且事件仍 OPEN/ACKNOWLEDGED/MITIGATING/VERIFYING：累加 alertCount；
- resolved：告警转 RESOLVED，关联事件转 RESOLVED；
- 之后再次 firing：创建新的事件，而不是复用已经解决/关闭的事件；
- 新事件使用新的 idempotency key，可以再次运行处置。

### 10.3 Runbook 路由

在 labels 或 annotations 中设置：

```yaml
paiops_runbook_id: "12"
paiops_auto_execute: "true"
```

`paiops_auto_execute=true` 不是无条件授权。后端解析 Runbook，要求：

- Runbook 存在；
- engineType 为 DAG；
- 节点非空；
- 所有节点风险等级均为 READ_ONLY。

只要包含通知、写入、审批、未知节点或 Kubernetes 动作，自动执行就不会发生，用户必须从事件中心手动启动。

## 11. 健康证据与事件状态

Runbook 成功不代表故障恢复。系统遍历节点输出，识别布尔字段：

```json
{"healthy": true}
```

决策：

| 任务结果 | 健康证据 | 事件结果 |
|---|---|---|
| 非 SUCCESS | 任意 | OPEN（WAITING_APPROVAL 保持处置过程） |
| SUCCESS | 至少一个，且全部 true | RESOLVED |
| SUCCESS | 任意一个 false | OPEN |
| SUCCESS | 没有 healthy 字段 | VERIFYING |

健康检查节点应尽量统一输出 `healthy`。普通查询或 AI 节点的自然语言结论不自动作为恢复证据，避免幻觉导致错误关闭。

## 12. 审计

审计记录至少覆盖：

- 登录和敏感配置变更；
- 任务入队、开始、完成、取消；
- 告警接入和事件更新；
- Runbook 从事件启动；
- 审批批准/拒绝；
- 凭证和 MCP 配置管理。

审计 detail 必须存脱敏后的结构化信息，不得写 access token、API Key、Authorization、密码、完整 kube bearer token。日志也不得打印 `.env` 内容。

## 13. 安全上线检查表

- [ ] `.env` 权限为 600；
- [ ] 所有 CHANGE_ME 均已替换；
- [ ] 管理员密码为随机强密码；
- [ ] JWT、Worker、Webhook、Master Key 各自独立；
- [ ] `PAIOPS_MCP_ALLOW_CUSTOM_COMMAND=false`；
- [ ] HTTP 白名单没有 `*`；
- [ ] Kubernetes ServiceAccount 不是 cluster-admin；
- [ ] 对外只开放前端 HTTP/HTTPS，MySQL/Redis/MinIO 控制台不开放；
- [ ] TLS 在反向代理或入口网关终止；
- [ ] 已备份 master key；
- [ ] 已执行 Dry Run；
- [ ] 已验证无审批的高风险动作被拒绝；
- [ ] 已确认配置查询不回显密钥；
- [ ] 已检查代理日志不含 JWT 查询参数。
